# C++ 面试题重点整理

## 1. 布隆过滤器（Bloom Filter）

### 核心结论
- 结构：位图（m bits）+ k 个哈希函数。
- 插入：对元素做 k 次哈希，对应位都置 1。
- 查询：
  - 只要有一位是 0 -> 一定不存在。
  - 全是 1 -> 可能存在（可能误判）。
- 性质：
  - 不会漏判（False Negative）
  - 会误判（False Positive）

### 公式（面试高频）
- 单个位仍为 0 的概率：`(1 - 1/m)^(kn) ≈ e^(-kn/m)`
- 误判率近似：`p ≈ (1 - e^(-kn/m))^k`
- 最优哈希个数：`k_opt = (m/n) ln 2`
- 给定 n 和 p 估算位图大小：`m ≈ -n ln p / (ln 2)^2`

### 工程落地（缓存穿透）
- 请求链路：先查 Bloom -> 命中再查缓存/DB。
- 删除难点：标准 Bloom 不支持安全删除。
- 常见方案：
  - 计数布隆过滤器（支持删，内存更大）
  - 周期重建 + 双版本切换
  - 软删除 + TTL

### 常见误区
- 误区：Bloom 能精确判断存在。
- 误区：哈希函数越多越好。
- 误区：忽略容量规划导致误判率失控。

## 2. 用户态线程（协程/Fiber） vs 内核线程

### 核心对比
- 内核线程切换：进入内核调度，保存/恢复更多上下文，开销更大。
- 协程切换：多数在用户态完成，不触发内核调度，切换更轻。

### 为什么协程更快
- 避免频繁内核态切换。
- 切换状态更少，路径更短。

### 代价转移到哪里
- 需要自己处理调度、公平性、饥饿、取消、超时。
- 若出现阻塞调用，可能卡住整个 worker 线程上的协程。
- 调试与可观测性复杂度更高。

### 选型经验
- 选“协程 + 少量内核线程”：高并发 I/O 密集服务。
- 选 1:1 线程：CPU 密集、阻塞库多、稳定性与工具链优先。

### 常见误区
- 误区：协程一定比线程快。
- 误区：用了协程就不会阻塞。

## 3. std::promise / std::future / std::async 错误传播

### 核心机制
- 共享状态（shared state）里保存“值”或“异常”。
- 生产者：`set_value` / `set_exception`。
- 消费者：`future.get()` 取结果；若是异常则在调用方线程重抛。

### API 语义要点
- `get()`：
  - 可能抛任务异常。
  - 对 `std::future` 通常只能调用一次。
- `wait()`：只等待，不抛任务异常。
- `valid()`：仅表示是否有共享状态，不代表任务成功。

### async 启动策略
- `std::launch::async`：并发执行，异常在 `get()` 时暴露。
- `std::launch::deferred`：延迟到 `get()/wait()` 才执行，异常也在那时暴露。
- 默认策略（async|deferred）存在实现差异，关键路径建议显式指定策略。

### 工程建议
- 小规模并发可用 `std::async`。
- 需要控制并发度/队列/背压时，优先线程池。
- I/O 密集异步链路，优先协程。

## 4. if constexpr、SFINAE、Concepts 的分工与边界

### 一句话区分
- 普通 `if`：运行时分支，两个分支都要通过编译期语义检查。
- `if constexpr`：编译期分支，未选分支在当前实例化中是 discarded statement。

### 关键边界
- `if constexpr` 未选分支并非“纯文本删除”，仍需语法合法。
- 依赖名错误常可被未选分支屏蔽；非依赖硬错误仍可能报错。

### 分工建议
- Concepts / requires：做“接口约束”（是否参与重载）。
- `if constexpr`：做“函数体内部策略分支”。
- SFINAE：老代码兼容场景可用，但新代码优先 Concepts。

### 约束重叠与意外匹配
- 常见问题不是二义性本身，而是语义意外：
  - `std::string`、字符串字面量可能匹配 range 分支。
- 解决方式：
  - 定义子概念 + 排他条件（推荐）
  - 或单入口 `if constexpr` 按优先级分派

### 面试可复述模板
- “我会先用 Concepts 缩小候选集，再在实现体内用 `if constexpr` 做细分，这样报错清晰、扩展成本低、避免约束重叠导致歧义或意外匹配。”

## 5. 动态库 ABI 稳定性与 pImpl

### 核心结论
- 即使只修改类的 `private` 成员，也可能破坏 ABI。
- 原因不是访问权限，而是对象布局、大小、成员偏移等二进制契约发生变化。
- 旧客户端二进制仍按旧头文件假设运行，新动态库按新布局执行，导致错配。

### 典型失效路径
- 栈上对象：客户端按旧 `sizeof(Foo)` 分配栈空间，新库按新布局写入，可能栈越界。
- `new Foo`：分配大小由调用点决定，旧客户端可能少分配，新库访问越界。
- 内联函数：成员偏移在客户端编译期固化，库升级后可能错读错写。

### 后果类型
- 崩溃（栈/堆越界触发）。
- 延迟性内存破坏（堆元数据损坏、随机崩溃）。
- silent data corruption（最隐蔽，结果错误但不立即崩）。

### pImpl 如何缓解
- 对外头文件仅暴露稳定壳对象（通常一个指针）。
- 可变实现细节放入 `Impl` 并隐藏在 `.cpp`。
- 这样 public ABI 边界更稳定，减少类布局变化对调用方二进制的冲击。

### pImpl 的代价
- 一次间接访问，缓存局部性变差。
- 可能多一次动态分配。
- 代码复杂度上升（拷贝/移动语义、异常安全、资源管理）。

## 6. 本轮速记版（30 秒）

- Bloom：可能误判，不会漏判；核心是 m、n、k 的平衡与工程重建策略。
- 协程 vs 线程：协程切换轻但治理复杂，阻塞治理是成败关键。
- future/async：异常存入 shared state，`get()` 重抛，`wait()` 不抛。
- 模板分支：Concepts 管入口，`if constexpr` 管实现，避免约束重叠与意外匹配。
- ABI 稳定性：`private` 不等于 ABI 私有；`sizeof/offset` 变了就可能炸，pImpl 用稳定句柄隔离变化。

## 7. 极简背诵版（每题 5 句）

### Q1 布隆过滤器（5 句）
- 布隆过滤器是位图 + 多哈希，用空间换查询速度。
- 查询为“全 1 则可能存在，有 0 则一定不存在”。
- 它会误判但不会漏判，前提是不用普通 Bloom 做删除。
- 误判率由 m、n、k 决定，k 不是越多越好，存在最优值。
- 工程上用于缓存穿透拦截，删除通常用计数 Bloom 或周期重建。

### Q2 协程/Fiber vs 内核线程（5 句）
- 内核线程切换要进内核调度，保存恢复状态更多，开销较大。
- 协程切换通常在用户态完成，切换更轻，适合高并发 I/O。
- 协程快的代价是调度、公平性、取消、超时都要自己治理。
- 一次阻塞调用可能卡住整个 worker 上的协程，是落地大坑。
- I/O 密集选“协程 + 少量线程”，CPU 密集或阻塞库重选 1:1 线程。

### Q3 promise/future/async 异常传播（5 句）
- 结果和异常都放在 shared state 中。
- 任务线程抛异常后会被捕获并存入 shared state。
- `wait()` 只等完成，不会抛任务异常。
- `get()` 读取结果并在调用方线程重抛异常。
- 关键路径不要依赖默认 async 策略，建议显式指定 `launch` 或改线程池/协程。

### Q4 if constexpr / Concepts / SFINAE（5 句）
- 普通 `if` 是运行时分支，两边都要通过编译检查。
- `if constexpr` 是编译期分支，未选分支在当前实例化中被丢弃。
- 但未选分支并非纯文本删除，语法错误或非依赖硬错误仍可能报错。
- Concepts 负责“能不能进这个重载”，`if constexpr` 负责“进来后怎么走分支”。
- 工程上先用 Concepts 缩小候选集，再用 `if constexpr` 做实现分派，可维护性最好。

### Q5 动态库 ABI 与 pImpl（5 句）
- 动态库 ABI 看的是二进制契约，不是源码访问权限。
- 改 `private` 也可能改变 `sizeof`、成员偏移和对象布局。
- 旧客户端按旧假设分配和访问，新库按新布局执行，会出现越界或错读。
- 风险包括崩溃、延迟性堆破坏和 silent data corruption。
- pImpl 把变化封装到实现侧，用稳定指针壳维持 ABI，但会引入一次间接和工程复杂度。
