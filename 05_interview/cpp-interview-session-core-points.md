# C++ 技术面试核心知识点汇总（本轮）

> 覆盖本次面试中出现的 10 个主题，按题目顺序归纳核心结论、工程要点与高频误区。

## 1. 系统调用开销与优化（syscall / vDSO）

### 核心机制
- 系统调用开销不只是“用户态/内核态切换”概念，而是具体硬件与内核路径代价叠加：
  - 特权级切换（Ring3 -> Ring0）
  - 内核栈切换
  - 寄存器保存/恢复
  - TLB/Cache 污染
  - 分支预测失配
- 在启用 KPTI 等安全补丁后，调用成本会进一步上升。

### 关键优化
- 优先减少 syscall 次数（批处理、合并调用、异步化）。
- 对时间获取等场景使用 vDSO 快路径（如 `clock_gettime` 常见时钟类型），避免频繁陷入内核。
- 高并发 I/O 可使用 `io_uring` 等机制降低系统调用密度。

### 常见误区
- 误以为每次 `clock_gettime` 都必然陷入内核。

## 2. 原子内存序（relaxed / acquire-release / seq_cst）

### 核心语义
- `memory_order_relaxed`：只保证原子性，不提供跨操作顺序保证。
- `memory_order_release`（写）+ `memory_order_acquire`（读）：建立发布-获取同步关系。
- `memory_order_seq_cst`：全局单一总序，语义最强、可理解性最好、开销通常最大。

### 关键结论
- Release/Acquire 必须配对才有同步意义。
- 当 acquire 读到 release 写入值时，release 前写入对 acquire 后读取可见（happens-before）。

### 常见误区
- 将 relaxed 误解为“弱顺序但仍有部分时序保证”。

## 3. False Sharing 与缓存一致性

### 本质
- 多线程写入不同变量，但变量位于同一 cache line，触发一致性协议频繁失效与转移（ping-pong）。

### 性能影响来源
- 失效后需要重新取 line（可能是 cache-to-cache 转移，也可能回内存），显著高于本地 cache hit。

### 工程规避
- 按 cache line 对齐热点写变量：`alignas(64)` 或 `std::hardware_destructive_interference_size`。
- 将高频写热点拆分，避免共享 line。
- 使用 `perf c2c` 等工具定位 cache line 争用。

### 常见误区
- 以为“必须回主内存读取”，实际常见是核间缓存传输。

## 4. CRTP 静态多态

### 机制
- 模式：`class Derived : public Base<Derived>`。
- 基类中通过 `static_cast<Derived*>(this)->impl()` 调派生行为。

### 性能差异（相对虚函数）
- 编译期分发，可内联，避免 vtable 间接跳转。
- 无 vptr/vtable 运行期开销。

### 适用边界
- CRTP 适合静态多态与 mixin 复用；
- 运行期异构对象统一处理仍需虚函数/类型擦除。

## 5. ABI 稳定性与 pImpl

### 为什么 private 变化也会破坏 ABI
- private 成员同样参与对象布局（size/offset/alignment）。
- 仅替换动态库不重编调用方时，旧布局假设与新库不一致可能导致崩溃或未定义行为。

### pImpl 解决思路
- 头文件只暴露稳定外壳（指针大小固定），实现细节放到 `Impl`。
- 改动内部成员不影响外部对象布局，提升二进制兼容性。

### 代价
- 一次额外间接访问；常见实现伴随堆分配开销。

## 6. SFINAE / enable_if / Concepts

### SFINAE
- 全称：Substitution Failure Is Not An Error。
- 模板实参替换失败时，不报硬错误，而是移除该候选继续重载决议。

### enable_if 用法
- 通过条件是否成立决定是否产生有效类型，从而启用/禁用模板候选。

### Concepts 改进点
- 约束表达更直接（可读性更高）。
- 诊断信息更聚焦（报错更友好）。
- 参与约束排序，重载决议语义更清晰。

### 常见误区
- 把 SFINAE 与 `if constexpr` 混为一谈。

## 7. 构造/析构期间虚函数调用

### 关键规则
- 构造基类阶段，动态类型视为当前构造层；vptr 指向当前层 vtable。
- 因此在基类构造函数中调用虚函数，不会分发到派生重写版本。
- 析构过程同理，随着层级回退，vptr 也回退。

### 本质原因
- 语言层面保证对象未完成构造时不调用尚未稳定的派生语义，避免未初始化成员被访问。

### 常见误区
- 误以为“vtable 还没创建”；实际 vtable 是编译期静态对象。

## 8. `std::unordered_map` 冲突与退化

### 冲突处理
- 典型实现为链式哈希（bucket + 链/节点序列）。

### 退化到 O(n) 的条件
- 大量 key 落同一 bucket（哈希分布差或被构造碰撞输入）。
- 负载因子过高导致链变长。

### 工程与安全实践
- `reserve` + 合理 `max_load_factor`。
- 关键场景使用质量更高哈希函数（必要时加入随机种子）。
- 面向外部不可信输入时警惕哈希碰撞 DoS 风险。

## 9. LRU 设计（O(1) get/put）

### 标准结构组合
- 双向链表维护访问新旧顺序。
- 哈希表实现 key 到节点的 O(1) 定位。

### 关键设计点
- 哈希表 value 应存“链表节点迭代器/指针”，而非仅存值。
- 节点建议存 `(key, value)`，淘汰尾节点时可 O(1) 回删哈希项。
- 通过 `splice` 等 O(1) 节点移动更新“最近访问”顺序。

### 常见误区
- 只用链表导致 `get` 退化为 O(n)。

## 10. epoll 的 LT 与 ET

### 语义差异
- LT（默认）：只要 fd 仍可读/可写，`epoll_wait` 会持续返回事件。
- ET：仅在状态变化边沿通知；若本次未读/写到 `EAGAIN`，可能长期不再收到提醒。

### ET 实战要点
- 必须使用非阻塞 fd。
- 读/写事件中循环处理直到 `EAGAIN/EWOULDBLOCK`。
- 妥善处理 `EINTR`、对端关闭、半包与写缓冲。
- 可结合配额与任务队列避免单连接长期占用事件循环。

### 常见误区
- 将“未读空”描述成“内核丢数据”；更准确是应用层未继续消费导致连接逻辑卡住。
